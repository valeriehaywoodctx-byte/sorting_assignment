Executive Summary
This empirical analysis explores the transition from elementary O(n^2) sorting algorithms, such as Bubble Sort, to more sophisticated Divide-and-Conquer paradigms, such as Merge Sort and QuickSort. Through rigorous benchmarking across five distinct data distributions, this study validates the theoretical O(n \log n) efficiency of these modern algorithms. The primary finding of this research is the overwhelming performance gap between Week 1 and Week 2 implementations. While elementary sorts exhibited exponential-time growth, Merge Sort and QuickSort maintained near-linear scalability, even as input sizes reached 10,000 elements. Notably, the optimized QuickSort implementation using randomized pivot selection and three-way partitioning consistently outperformed Merge Sort by approximately 20-40% across most scenarios, particularly excelling on datasets with high redundancy (many duplicates). These results confirm that while both algorithms share the same theoretical complexity class, the constant-factor optimizations and spatial efficiency of QuickSort provide a distinct advantage in practical, localized execution environments.

Methodology
To ensure the empirical results of this study are both reproducible and scientifically valid, a rigorous benchmarking environment was established. Following the theoretical frameworks for Divide-and-Conquer algorithms established by Cormen et al. (2022), the testing suite was designed to measure execution time across various input distributions.
All benchmarks were executed on a local workstation featuring a Dell 16 Plus, Windows 11 OS, 32GB LPDDR5X RAM, and an Intel Core Ultra 7 258V. The environment was standardized using a Python 3.14 virtual environment to isolate the project dependencies. Automated validation was performed using the pytest framework to ensure that optimizations, such as the insertion sort threshold in QuickSort, did not compromise the correctness of the sorting logic.

As noted in Introduction to Algorithms (Cormen et al., 2022), the running time of an algorithm on a particular input is the number of primitive operations executed. To capture this empirically, this study utilized Python’s time.perf_counter(). This function was selected because it provides a fractional second clock with the highest available resolution, specifically designed for measuring short durations and avoiding the jitter common in standard system time.
In order to test the worst-case and average-case scenarios discussed in Cormen et al. (2022), five data types were generated:
Random: To observe average-case behavior.
Sorted/Reverse: To test the efficiency of the randomized pivot selection.
Nearly Sorted: To simulate real-world data where only 5% of elements are misplaced.
Many Duplicates: To validate the efficiency of three-way partitioning when the cardinality of the input is low.

Merge Sort Implementation
The implementation of Merge Sort in `src/sorting/merge_sort.py` follows the classic Divide-and-Conquer paradigm as described by Cormen et al. (2022). The algorithm recursively divides the unsorted list into \( n \) sub-lists, each containing just one element (the base case). It then repeatedly merges these sub-lists to create new sorted sub-lists until only one sorted sub-list remains. The core of this implementation is the merge helper function. As Cormen et al. (2022) explained, the merging step is crucial to the algorithm's efficiency. My implementation achieves a time complexity of \( O(n + m) \) by using two pointers to compare the heads of each sub-list and appending the smaller value to a new result list. This approach ensures that the original input remains unaltered, thereby adhering to the assignment's immutability requirement. While Merge Sort guarantees a worst-case time complexity of \( O(n \log n) \), it comes with a trade-off in terms of space complexity. Unlike in-place sorting algorithms, Merge Sort requires \( O(n) \) additional memory to store the subarrays during the merging process. In my testing, this was not an issue for input sizes up to 10,000, but it is an important consideration in memory-constrained environments.

Quicksort Implementation and Optimization
QuickSort is often faster in practice than Merge Sort due to its smaller constant factors, but it is notoriously sensitive to the choice of the pivot. To make this implementation production-ready, three specific optimizations were included. According to Cormen et al. (2022), the "worst-case" for QuickSort occurs when the pivot consistently results in highly unbalanced partitions (e.g., 0 and n-1). This often happens with already-sorted data. By using random.choice(arr), my implementation ensures an average-case balance regardless of the input order, thereby maintaining an expected O(n \log n) runtime. To address the Many Duplicates requirement, I implemented three-way partitioning. Instead of just less-than and greater-than, the algorithm creates an intermediate list containing all elements equal to the pivot. This allows the algorithm to sort a list of identical values in a single pass (O(n)), rather than recursively processing them. A common optimization in high-performance libraries is switching to a simpler algorithm for small subarrays. My implementation uses a threshold of 10. For lists smaller than this, the overhead of recursion outweighs the benefits of QuickSort. In these cases, the algorithm switches to Insertion Sort, which has lower overhead for tiny datasets.

Results & Performance Comparison
The empirical data collected in the benchmarks/week2_performance.py script confirms the theoretical superiority of O(n \log n) algorithms over the O(n^2) algorithms from Week 1. At an input size of 10,000, the performance gap was staggering. While a standard Bubble Sort would take several seconds to process 10,000 items, Merge Sort completed the task in approximately 0.011s, and QuickSort in 0.009s. This represents a speedup factor of over 1000x. As Cormen et al. (2022) noted, as n increases, the difference between these complexity classes becomes the single most important factor in software performance. 
The most notable result was the Many Duplicates scenario. QuickSort finished 10,000 items in 0.0019s, significantly faster than Merge Sort. This empirically validates the decision to use three-way partitioning, as the algorithm could skip sorting the middle elements once they were grouped.

Critical Analysis and Surprising Findings
One of the most compelling observations during this study was the resilience of the optimized QuickSort implementation. Traditionally, QuickSort is vulnerable to adversarial data, such as already-sorted or reverse-sorted lists. However, as established by Cormen et al. (2022), introducing a randomized pivot effectively mitigates this risk. In my empirical tests, the execution time for Sorted data (0.00736s) was nearly identical to that of " Random" data (0.00939s) at the 10,000-element mark, proving that the $O(n \log n)$ average-case was maintained.
Another surprising finding was QuickSort's efficiency on the Many Duplicates dataset. The implementation outperformed Merge Sort by nearly a factor of 5. This is attributed to the three-way partitioning logic, while Merge Sort continued to divide and merge the redundant values, the QuickSort partitioner grouped all identical keys into the middle array in a single pass, drastically reducing the depth of the recursion tree. This highlights a critical practical recommendation: when data cardinality is low (many duplicates), an optimized QuickSort is significantly more efficient than a standard Merge Sort.

Conclusion
This assignment provided comprehensive validation of the Divide-and-Conquer algorithms presented in Cormen et al. (2022). The empirical evidence gathered demonstrates that for modern computing tasks, the transition from O(n^2) to O(n \log n) algorithms is not merely a theoretical improvement but a practical necessity. While Merge Sort offers high consistency across all data types and is preferred when stability is required (preserving the relative order of equal elements), QuickSort remains the superior choice for raw performance in general-purpose applications. The inclusion of randomization, three-way partitioning, and an insertion sort threshold successfully addressed the inherent weaknesses of the standard QuickSort algorithm. Ultimately, these tests confirm that the mathematical abstractions used in theoretical computer science accurately predict software behavior in real-world environments.
